LCCN Harvester Project

Project year: 2025-2026

Client: Melissa Belvadi, Librarian at UPEI

[<u>mbelvadi@upei.ca</u>](mailto:mbelvadi@upei.ca)

Goal of the project: to create a GUI app that takes in a list of ISBNs
and attempts to find corresponding Library of Congress Call Numbers
(LCCNs) ([<u>MARC
050</u>](https://www.loc.gov/marc/bibliographic/bd01x09x.html)) given
specific API and related targets that can sometimes provide that
information. User may select to also get NLM Call Numbers(NLMs) (MARC
060). The rest of this document will refer to LCCNs, but everything also
applies to the NLMs. Users should be able to select in a saved
configuration whether they want either or both.

**MVP:**

The working title for this project will be "LCCN Harvester", or for the
rest of this document, "the harvester".

The harvester will use a combination of a few major APIs (Library of
Congress, Harvard, possibly one more) and an API protocol called z39.50
that can retrieve data from a long list of Z39.50 servers (other library
catalogues).

It should be built entirely using Python 3 and PyQT6 (no C, javascript,
etc.), and be cross-platform compatible with Windows, Mac and Linux.

All students must agree to the assignment of the MIT Open Source
License, as the final code will be freely shared with librarians around
the world. It should be delivered via Github repository, so the client
can easily clone it into her own Github account.

The input will be a simple list of ISBN numbers in a file that the user
will select using a standard file-picker dialogue for their platform.

There will be two forms of output for each lookup of each individual
ISBN.

The first is to write to an sqlite database the found data or failed
attempts.

The second is to write to a tab delimited file, which will have
<span class="mark">at most these columns (depending on the user's choice
of LCCN, NLM, or both):</span>

- <span class="mark">ISBN</span>

- <span class="mark">LCCN</span>

- <span class="mark">LCCN source</span>

- <span class="mark">NLM</span>

- <span class="mark">NLM source</span>

- <span class="mark">date</span>

Note that if they want both you can't write out the tsv row until you
found both or exhausted the target list.

<span class="mark">Let the user decide on a per "run" basis if they want
just LCCN, just NLM, or both. If the user chooses to try to collect
both, ask them:</span>

> <span class="mark">If you get just one of LCCN and NLM from one of
> your early "targets":</span>
>
> <span class="mark">\[ 1\] stop searching that ISBN when I get an LCCN
> only</span>
>
> <span class="mark">\[ 2\] stop searching that ISBN when I get an NLM
> only</span>
>
> <span class="mark">\[ 3\] stop searching that ISBN if I get either one
> first</span>
>
> <span class="mark">\[ 4\] keep going through the targets list stopping
> only if I got both or have exhausted the target list</span>

<span class="mark">If the user picks option 1, 2, or 3, do NOT record
the other one as a failure, because we didn't actually look exhaustively
for the second one.</span>

<span class="mark">If the user picks option 4, it is fine if they find
them in two separate targets, eg they get the LCCN only in the Harvard
API, but then they get the NLM from one of the Z3950 targets.</span>

A third file will be generated if any of the ISBNs are invalid, just
listing those.

In each case when processing a new list of isbns, each ISBN should be
looked up in the existing sqlite table first to reuse what was
previously found before attempting either an API or z39.50 lookup. If
the ISBN is not found in the main sqlite data table, the harvester
should also check the failure table (see description below) before
attempting any lookups based on the the user-selected period of time to
not retry (see below - would be scale of days). Only after checking the
main data table and the failure table should the harvester then proceed
to check each of the apis and z39.50 sources in order to find a match.

The harvester should accept ISBN numbers formatted with or without
hyphens and normalize that data to having no hyphens for both forms of
output. It should accept both 10 digit and 13 digit isbns and treat
isbns always as text and never lose any possible leading zeros that can
occur on 10 digit isbns. It should also do a basic check-digit
validation to catch invalid ISBNs. Invalid ISBNs will be written to a
simple text file.

The harvester will also need to validate the Library of Congress Call
Numbers that it finds against some basic rules for properly formed
LCCNs, as there are many instances in which the API target returns an
invalid LCCN.

Call numbers should be normalized by using the value of the \$a and
replacing \$b with a space.  
For instance, an LCCN that appears in the MARC as:  
050 10\$aHF5726\$b.B27 1980

will get normalized by converting the \$b to a space, resulting in  
HF5726 .B27 1980

LCCNs always start in the 050 with a \$a. If there is more than one \$a,
like this example, only use the first:  
050 00\$aZ7164.N3\$bL34 no. 9\$aZ7165.R42\$aHC517.R42

becomes just: Z7164.N3 L34 no. 9

This is the same rule for NLMs.

There is open source code for the z39.50 called
[<u>Yaz</u>](https://www.indexdata.com/resources/software/yaz/) client
but it's written in C and part of this project may be to convert that
into Python 3 format, unless the students decide it would be easier to
create their own python module for z39.50 from scratch. The actual C
code may not be used - this harvester must be entirely in python.

All of the data that will go into the sqlite database for the harvester
will go into one of two tables. The "main" table will have the following
columns:

- ISBN number which will be type varchar(13) (and will be stripped of
  > hyphens),

- <span class="mark">LCCN or NLM call number which will be of type
  > varchar(64),</span>

- <span class="mark">Which kind of call number is stored in that
  > previous column, "NLM" or "LCCN", type varchar(4)</span>

- Library of Congress classification which will be the first one, two,
  > or three letters of the Library of Congress call number, varchar(3)

- the source of the LCCN/<span class="mark">NLM</span> found; that is,
  > which API or which z39.50 catalog it came from (varchar(64)

- the date added/updated in format yyyymmdd (integer)

So if both LCCN and NLM are requested and found, there would be two rows
in this table for that ISBN.

The second table "attempts" will be for recording failed attempts and
will consist of the following columns:

- <span class="mark">which of LCCN/NLM this row is for,</span>

- <span class="mark">the ISBN number (stripped of hyphens)
  > attempted,</span>

- <span class="mark">the API/target attempted, and</span>

- <span class="mark">the date attempted.</span>

- <span class="mark">\[added march 5, 2026\] Reason it failed if not
  > just "no match"</span>

> <span class="mark">So for instance, if neither is found, there would
> be two rows for that ISBN.</span> The principle is that if the user
> tries and fails to look up that ISBN within a fixed (set by the user)
> number of days since the last attempt on that ISBN and that target,
> the tsv output will have the failure info without attempting those
> target sources. If the user has added a new Z39.50 target, though, the
> target will be tried even though there are failures in that table from
> other targets.

<span class="mark">If both are requested but only one is found, the
found one gets a row in the main table, and the not-found one gets a
line in the "attempts" table.</span>

<span class="mark">If the user requests both LCCN and ISBN one day and
one of them is not found (added a row to attempted) and tries that one
again after their selected "number of days to try again" and finds it,
the row would be deleted from "attempts" and added to "main". There
should never be a row in "attempts" for a given ISBN and call number
type, when there is a "main" row for that same pairing.</span>

The non-Z39.50 APIs to be included are:

- Library of Congress API:
  > [<u>https://www.loc.gov/apis/json-and-yaml/</u>](https://www.loc.gov/apis/json-and-yaml/)

- Harvard API:
  > [<u>https://wiki.harvard.edu/confluence/display/LibraryStaffDoc/LibraryCloud+APIs#LibraryCloudAPIs-AbouttheItemsAPI</u>](https://wiki.harvard.edu/confluence/display/LibraryStaffDoc/LibraryCloud+APIs#LibraryCloudAPIs-AbouttheItemsAPI)

- OpenLibrary API:
  > [<u>https://openlibrary.org/developers/api</u>](https://openlibrary.org/developers/api)

In addition, the user will be able to maintain (add, modify, delete) a
table of Z39.50 "targets" to check. The MVP will include a 'starter'
table of some large common targets (to be provided by the client), and
will include the proper settings required for Z39.50 calls.

The users will be able to select and rank the order of all targets, APIs
and Z39.50s, for any given run, and save their preferences for default
load the next time they run the harvester, either in the same "session"
or a new one. <span class="mark">If the user selects to try to collect
LCCN and NLM, at that point pop up something to ask them (sample
text):</span>

> <span class="mark">If you get just one of LCCN and NLM before you have
> exhausted your selected list of "targets":</span>
>
> <span class="mark">\[ \] stop searching that ISBN when I get an LCCN
> only</span>
>
> <span class="mark">\[ \] stop searching that ISBN when I get an NLM
> only</span>
>
> <span class="mark">\[ \] keep going through the targets list stopping
> only if I got both or have exhausted the target list</span>

Once the harvester finds a valid call number <span class="mark">or
numbers given the option selected above</span>, it stops there and does
not try to find it/them again in another target. The user can on any run
deselect certain targets so the run is only on the targets they
selected, and in the order they selected. If the user changes the order
of either target group (APIs and Z39.50 sites), they should be asked if
they want to save that selection and order for the next use. This means
that the files these are saved in will also have a field "selected"
(on/off) that will be the default for the next run in this or another
use of the harvested. Unselected targets will still be available in the
GUI for future selection, but appear "greyed out"

The harvester should provide both an option within the GUI to manually
add individual z39.50 targets and also maintain that data in a simple
tab delimited file that can be easily edited directly by the user. The
non-Z39.50 API targets' data will be in the same file, so the user's
rank preferences across both types can be saved. The documentation
should include a link to one or more Z39.50 sites where that information
can be found; those links will be provided by the client. In addition to
an integer column for rank, there will also be a "selected" on/off
column, which will load as the defaults the next time the harvester is
run.

It is not expected that a user will be able to add another non-Z39.50
API as each of those will likely require custom code. The structure of
the tab delimited "targets" file will need to take into account that
there will be different metadata/settings needed for each kind of
target. Z39.50 targets will be relatively uniform but the other APIs
will have some unique requirements. If the students are unable to make
them all work within a single file, having separate files will be
acceptable but they must take into account how to provide the user's
ranks across them.

**Stretch Goals (in order from most to least desirable):**

    > files on the local drive. Prompt the user for what to use as the
    > "source" of the LCCN. These will go into the same tables as the
    > rest.

2.  Add a fourth table to the sqlite database, that links together
    > multiple ISBNs assigned to the same book. It must be the same
    > "edition" in the textbook numbered sense (8th, 9th, etc.) but it
    > can be in various formats, eg print hardcover, print paperback,
    > eISBN from one platform, eISBN from another, etc.

    1.  use the lowest number as the primary one for the main and
        > attempted tables  
        > Note that "lowest" needs to deal with ISBNs whose last digit
        > is actually a letter - treat that letter as a "9" for this
        > purpose but don't actually change it to a "9")

    2.  update the main and attempted with the lowest if you find a
        > lower one later

    3.  the rows in this table are going to be two columns: lowest,
        > other. So you might have these rows for a book that has three
        > different ISBNs found:  
        > 1234567890 9781234567890  
        > 1234567890 9784312567891  
        > and for the main table, only 1234567890 would appear,
        > regardless of which of the three was found first

    4.  even if the lowest ISBN is not found on the specific record
        > where the LCCN was found, the main table would still have the
        > lowest ISBN with an LCCN that came from a record that had a
        > higher paired ISBN

        > (thesauri)</u>](https://www.loc.gov/marc/bibliographic/bd648.html)

        > ind for each of the 6XXs separately, just one set of 2nd-ind
        > preferences across all of their 6xx preferences

    6.  if the record has more than one of the desired 6xx fields, use

Appendix:

The MARC fields, indicators, and subfields are defined at:

[<u>https://www.loc.gov/marc/bibliographic/</u>](https://www.loc.gov/marc/bibliographic/)

The Library of Congress Call Number system is defined at:

[<u>https://www.loc.gov/aba/publications/FreeCSM/G055.pdf</u>](https://www.loc.gov/aba/publications/FreeCSM/G055.pdf)

The NLM system is structurally the same, just with different letters and
numbers.

The painful details are at:
[<u>https://www.oclc.org/bibformats/en/0xx/060.html</u>](https://www.oclc.org/bibformats/en/0xx/060.html)

The basics of ISBNs are explained well here:
[<u>https://en.wikipedia.org/wiki/ISBN</u>](https://en.wikipedia.org/wiki/ISBN)
